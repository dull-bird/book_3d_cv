# 本书写作风格指南

## 核心原则：把读者当初学者

- 每个概念从零开始解释，不假设读者已经知道任何前置术语。
- 先给直觉，再给公式，最后给代码——三步走。
- 公式之后必跟"人话翻译"。
- 代码注释要解释"为什么"，不只是"做什么"。

## 比喻的使用

- 可以有比喻，但不要太过分。
- 比喻的目的是帮助理解，不是娱乐。
- 每个比喻后面要回扣到精确的数学/物理概念。
- 禁止为了俏皮而牺牲准确性。

## 知识讲解的四问

每个重要知识点必须回答：

1. **是什么？** —— 定义和直观理解
2. **有什么用？** —— 实际应用场景
3. **解决了什么问题？** —— 如果没有它，会发生什么
4. **有什么局限？** —— 什么时候失效、边界条件

## 补充说明风格（三层体系）

全书统一三种视觉层次，禁止自创其他 callout 类型：

**L1 `[!CAUTION]` —— 警告（红色）**
- 只用于会导致错误结果、系统性偏差、踩坑的场景
- 当前全书保留 3 个：物理焦距 vs 像素焦距混淆、呼吸效应导致标定失效、tilted model 误用

**L2 `[!NOTE]` —— 概念（蓝色）**
- 需要读者停下来理解的重要概念澄清
- 当前全书保留 5 个：章节导读、景深澄清、虚拟成像平面、K 矩阵像素焦距解释、本节不讲标定

**L3 `[!TIP]` —— 公式解释 / 人话翻译（紫色）**
- 每个重要公式后必须跟一个人话翻译，用 `> [!TIP]` 包裹
- mdbook 0.5.2+ 原生支持 GitHub-style alerts（`[!TIP]` / `[!NOTE]` / `[!CAUTION]`），无需任何插件
- 样式通过 `theme/css/custom.css` 覆盖为紫色背景块（`.blockquote-tag-tip`）
- 内容中不要加 `**人话翻译**：` 前缀，直接写解释文字

**L4 `> **补充说明**：` —— 补充（灰色引用块）**
- 技术细节、副作用、背景知识等不需要 callout 强调的补充内容

**禁止**
- 不要把普通说明硬塞进 `[!CAUTION]` 或 `[!WARNING]` 来刷存在感
- 不要在正文中用 `**粗体标题**` 做补充说明（要么升成 L3 引用块，要么直接融入正文）

## 章节结构

每节遵循以下结构：

1. **前置知识** —— 告诉读者需要已经掌握什么
2. **第一阶：直观理解** —— 场景、直觉、技术全景
3. **第二阶：原理解析** —— 推导、公式、数学
4. **第三阶：部署实战** —— 代码、API、常见陷阱
5. **苏格拉底时刻** —— 思考题，检验理解
6. **延伸阅读** —— 文献和本书内相关章节

## 公式与代码

- 每个重要公式后要有"人话翻译"。
- 变量命名要直观，避免单字母（除非数学惯例）。
- 代码块要有输入/输出示例，让读者能直接复制运行。

## 图片使用

### 图片来源优先级
- 优先从 H&Z 原书 PDF 精确截取。
- 自制图或外部来源图仅作补充，禁止用 AI 生成示意图充数。

### 目录与命名规范
- **H&Z 原书图**：统一放 `assets/figures/hz_hires/`，命名格式 `fig{章节}_{描述}.png`
- **自制图 / 外部图**：按章节建子目录，如 `assets/figures/01_camera_model/`、`assets/figures/02_coordinates/`
- **格式**：统一使用 `.jpg` 或 `.png`，禁止 `.avif`、`.webp` 等浏览器兼容性差的格式
- **引用路径**：`../assets/figures/{子目录}/{文件名}`
- **样式**：图片默认全局居中（通过 `theme/css/custom.css` 控制），不要在单个 markdown 中手动写 HTML 居中

### 图注规范
- 每张图要有明确的图注，说明来源和含义。
- 示意图用 ASCII 或 mermaid，照片用实际图像。
- 多模态提取图片时要验证准确性。

## 术语统一

- 同一术语全书统一，不混用中英文。
- 首次出现术语时给出英文原文。
- H&Z 引用格式统一：`H&Z section X.Y, p.ZZZ`
